## Wat LaventeCare is
LaventeCare is a single-tenant, Dutch-language B2B CRM + operations + billing application embedded in the JeffriesBackend monorepo, built for Jeffrey's care-sector ("zorg") consultancy (LaventeCare, KVK 88162710, Dronten, jeffrey@laventecare.nl). It is single-tenant in the strictest sense: every lc_* table carries user_id TEXT NOT NULL with no FK to a users table, and all data is user-scoped. It spans the full commercial lifecycle: a customer dossier (lc_companies) as the hub, a sales funnel (signal → lead → project, with a parallel lighter workstream/"opdracht" track), a billing stack (quote → time entries → invoice with bunq payment requests and UBL/HTML invoice documents), governance objects (decisions, change requests, SLA incidents), a knowledge-document catalogue plus per-context generated PDF dossiers, an outbound Microsoft-Graph customer mailbox with AI-assisted drafting, and an activity timeline. It is surfaced three ways: a REST API under /api/v1/laventecare (consumed by a separate Next.js frontend repo), a dedicated "laventecare" AI agent exposing 29 tools (10 read + 19 mutating, all mutations confirmation-gated), and Telegram (the /laventecare or /lc command, free-text intent routing, and a read-only /focus/summary dashboard). Google (Gmail + Calendar) feeds a business-signal scanner and business-context tagging; Microsoft Graph powers the branded client mailbox.

## Entiteitenmodel
- **lc_companies (LCCompany)** — Central CRM hub = customer/relation dossier (klantdossier). Carries CRM fields (naam, website, sector, status default 'prospect', relatie_type default 'prospect', notities, laatste_contact, volgende_actie) plus a business-core block added via ALTER: kvk_number, vat_number, billing_email/address/reference, payment_terms_days (default 14), contract_status (default 'geen_contract'), service_level (default 'basis'), preferred_channel, portal_url, default_login_url, onboarding_status (default 'niet_gestart'), data_processing_status (default 'niet_nodig'). Struct also exposes non-persisted counts (contacts/leads/workstreams/projects/open actions/dossier docs).  
  _links:_ Referenced by almost every child table. ON DELETE SET NULL everywhere EXCEPT lc_access_credentials.company_id and lc_activity_events.company_id which are NOT NULL + ON DELETE CASCADE. DeleteCompany is a GDPR Art.17 erasure: tx deletes lc_contacts explicitly, NULLs business_context on notes + personal_events, then deletes the company (cascade clears creds + timeline).
- **lc_contacts (LCContact)** — Contact persons. Fields: company_id, naam, email, telefoon, rol, is_primary (default false), notities, preferred_channel, decision_role.  
  _links:_ company_id → lc_companies (SET NULL). Setting is_primary=true clears other primaries on the same company (clearOtherPrimaryContacts). Also referenced by leads.contact_id, access_credentials.contact_id, activity_events.contact_id, mail_outbox.contact_id. Treated as PII — explicitly deleted on company erasure.
- **lc_leads (LCLead)** — Top-of-funnel sales lead. Fields: company_id, contact_id, titel, bron (default 'cockpit'/'ai'), source_id, status (start 'nieuw'), fit_score, pijnpunt, prioriteit (default 'normaal'), volgende_stap, volgende_actie_datum. LCLeadCreate has transient company_name/website for on-the-fly company creation.  
  _links:_ company_id/contact_id → companies/contacts (SET NULL). project.lead_id back-links the originating lead. De-dup on (user,bron,source_id) via GetLeadBySource. ConvertLeadToProject sets status='gewonnen'.
- **lc_projects (LCProject)** — Delivery engagement. Fields: company_id, lead_id, naam, fase (default 'intake'), status (default 'actief'), waarde_indicatie, start_datum, deadline, samenvatting.  
  _links:_ company_id/lead_id → companies/leads (SET NULL). Target of both lead→project and workstream→project conversions. Parent of decisions, change_requests, sla_incidents.
- **lc_workstreams (LCWorkstream / opdracht)** — Flexible lighter work track for small/medium jobs. Rich fields: titel, type (default 'advies'; e.g. website_platform/integratie/automatisering/ai_workflow/crm_sales/data_reporting/security_privacy/support_beheer/discovery_advies), status (default 'nieuw'), prioriteit, company_id, lead_id, project_id, klant_naam, bron, source_id, doel, scope, deliverable, bevindingen, volgende_stap, deadline, geschatte_minuten, waarde_indicatie, stack_tags[], tags[], completed_at.  
  _links:_ company_id/lead_id/project_id → SET NULL. If linked to a project the customer must match. completed_at set when status becomes closed. ConvertWorkstreamToProject sets status='omgezet_project'.
- **lc_access_credentials (LCAccessCredential)** — Encrypted customer portal/logins. Fields: company_id (NOT NULL), contact_id, project_id, workstream_id, title, login_url, username, role, environment (default 'pilot'), status (default 'actief'), owner_contact, secret_label (default 'wachtwoord'), secret_value_encrypted (AES-GCM, uitsluitend met de dedicated LAVENTECARE_SECRET_KEY; writes blijven uit bij een ontbrekende/zwakke key), secret_hint, sharing_policy (default 'veilig_kanaal'), last_checked_at, expires_at, revoked_at, notes. API never returns the secret — only secret_configured bool + secret_hint.
  _links:_ company_id NOT NULL → companies ON DELETE CASCADE (the only mandatory+cascade child). contact/project/workstream_id → SET NULL. Status → 'ingetrokken'/'verlopen' sets revoked_at; count ignores 'verwijderd'.
- **lc_action_items (LCActionItem; create-type LCActionCreate)** — Follow-up tasks. Fields: source (default 'handmatig'/'ai'), source_id, title, summary, action_type (default 'opvolgen'), status (start 'open'), priority (default 'normaal'), due_date, plus polymorphic links linked_lead_id/linked_project_id/linked_workstream_id/linked_company_id (ALTER). NB table is lc_action_items, not lc_actions.  
  _links:_ All four linked_* FKs → SET NULL. List shows only status in (open,bezig,wacht_op_klant) sorted by due_date. activity_events.action_item_id → action items.
- **lc_activity_events (LCActivityEvent)** — Customer-dossier timeline. Fields: company_id (NOT NULL), contact_id, lead_id, project_id, workstream_id, action_item_id, event_type (default 'notitie'), channel (default 'manual'), title, body, occurred_at (default now()). Struct carries join-name fields.  
  _links:_ company_id NOT NULL → companies ON DELETE CASCADE; other links SET NULL. event_types contact/gesprek/call/meeting/afspraak/email update company.laatste_contact. Sent template mail auto-logs an 'email' event. time_entries.activity_event_id links hours to events.
- **lc_documents (LCDocument)** — Global seeded knowledge-base / template catalogue (not customer-bound). Fields: document_key, titel, categorie (commercieel/proces/governance), fase, versie (default '2026-04'), source_path, samenvatting (NOT NULL), tags[]. UNIQUE (user_id, document_key).  
  _links:_ SeedDocuments bulk-upserts on conflict. Scored by BuildDossierAdvice to produce recommendations. Matched catalogue→dossier on document_key.
- **lc_dossier_documents (LCDossierDocument)** — Per-context generated PDF history (distinct from catalogue). Fields: document_key, titel, template_label, context_type (default 'manual'), context_id, context_title, lead/project/workstream/company links, pdf_url (NOT NULL), theme (default 'screen'), delivery (default 'inline'), notes, generated_at. No updated_at.  
  _links:_ lead/project/workstream/company_id → SET NULL; company_id derived from lead/workstream/project if omitted. All links validated company-scoped.
- **lc_decisions (LCDecision)** — Project decision log / audit trail. Fields: project_id, titel, besluit (required), reden (default 'Niet gespecificeerd'), impact, status (default 'genomen'), datum (default today). No updated_at.  
  _links:_ project_id → projects (SET NULL). Status patchable via /decisions/{id}/status.
- **lc_change_requests (LCChangeRequest)** — Scope/planning/budget change requests. Fields: project_id, titel (required), impact (required), planning_impact, budget_impact, status (default 'nieuw').  
  _links:_ project_id → projects (SET NULL). Open-list = status in (nieuw,beoordeeld,goedgekeurd).
- **lc_sla_incidents (LCSlaIncident)** — Support/SLA incidents. Fields: project_id, titel (required), prioriteit (default 'P3'), status (default 'open'), kanaal (default 'telegram'), gemeld_op (default now()), reactie_deadline, samenvatting.  
  _links:_ project_id → projects (SET NULL). Open-list = status in (open,in_behandeling,wacht_op_klant).
- **lc_quotes (LCQuote) + lc_quote_lines (LCQuoteLine)** — Quotes/offertes. quote_number LC-OFF-YYYY-NNNN (tx-allocated via advisory lock + MAX+1), titel, status (default 'concept'), issue_date, valid_until, currency EUR, subtotal_cents, vat_rate_bps (default 2100 = 21%), vat_cents, total_cents, accepted_at, notes. Lines: description, quantity (default 1), unit_amount_cents, total_cents, sort_order. UNIQUE (user_id, quote_number).  
  _links:_ company_id/project_id/workstream_id → SET NULL. quote_lines.quote_id → quotes ON DELETE CASCADE. Quote status='geaccepteerd' is prerequisite for invoice creation.
- **lc_time_entries (LCTimeEntry)** — Time/hours registration. Fields: company_id, project_id, workstream_id, activity_event_id, invoice_id, source_type (default 'manual'), source_id, description, entry_date, minutes (>0 required), hourly_rate_cents (default 7500 = €75/h), billable (default true), status (default 'concept').  
  _links:_ company/project/workstream/activity_event_id → SET NULL. invoice_id is a deferred FK (INITIALLY DEFERRED) → invoices SET NULL. On invoicing, invoice_id set + status → 'gefactureerd'. Company derived from project/workstream/activity.
- **lc_invoices (LCInvoice) + lc_invoice_lines (LCInvoiceLine)** — Invoices. invoice_number LC-FAC-YYYY-NNNN (tx-allocated), status (default 'concept'), issue_date, due_date (default issue+14d), currency, subtotal/vat/total/paid_cents, payment_provider (default 'bunq'), provider_request_id, merchant_reference, payment_url, document_url, ubl_xml, payment_checked_at/status/last_error, reminder_count, last_reminder_at, sent_at, paid_at, notes. UNIQUE (user_id, invoice_number) + partial UNIQUE (user_id, quote_id) WHERE status<>'geannuleerd' (max one active invoice per quote). Lines: source_time_entry_id, description, quantity_minutes, unit_amount_cents, vat_rate_bps, total_cents, sort_order.  
  _links:_ company/project/workstream/quote_id → SET NULL. invoice_lines.invoice_id → invoices ON DELETE CASCADE; line.source_time_entry_id → time_entries SET NULL. BTW summed per-line (mixed-rate correct); hour-line totals server-derived (minutes×rate).
- **lc_mail_templates (LCMailTemplate)** — Email templates (13 branded defaults via brandedMailHTML/seed). Fields: template_key, name, category (default 'general'), status (default 'active'), subject_template, body_html, body_text, default_cc[], default_bcc[]. UNIQUE (user_id, template_key).  
  _links:_ Referenced by mail_outbox.template_id (SET NULL).
- **lc_mail_outbox (LCMailOutboxItem)** — Sent/draft emails (outbound only — no inbound ingestion/threading). Fields: template_id, company/contact/project/workstream/quote/invoice_id, to_email (NOT NULL), to_name, cc[]/bcc[], subject, body_html, body_text, status (concept/sending/sent/failed/sent_unconfirmed), provider (default 'microsoft_graph'), provider_message_id, error_message, sent_at.  
  _links:_ Optional FKs to nearly every CRM/billing entity for mail context, all SET NULL. Sent via Microsoft Graph; MarkSent scrubs pilot passwords at rest.
- **Non-table DTOs / aggregates** — Operational/aggregate types with no table: LCCockpit/LCCockpitSummary (dashboard), LCBilling/LCBillingSummary, LCMailbox/LCMailboxSummary, LCBusinessSignal & LCFollowUpSignal (detected signals), LCDossierAdvice chain (read-only AI advice), conversion DTOs LCConvertLeadToProject / LCConvertWorkstreamToProject / LCConvertSignalToLead, LCMailAIContext/LCMailAISuggestion, LCSeedResult.  
  _links:_ Computed by GetCockpit, GetBilling, BuildDossierAdvice, buildBusinessSignals, buildFollowUps over the persistent tables.

## Kern-workflows
### Sales funnel: signal → lead → project
buildBusinessSignals (in GetCockpit) scans the last ~120 emails, personal_events (14d) and active notes against CRM-derived terms (company/lead/project/workstream names) + static markers (discovery, blueprint, voorstel, sla, verwerkersovereenkomst); unread matched emails = urgency 'hoog' (caps: email 8, agenda 6, notitie 8, total 18). ConvertSignalToLead (POST /signals/convert-lead) turns a signal into a lead (pijnpunt=subtitle+action_hint, prioriteit=urgency), de-duped on (bron,source_id) returning reused:true if it exists. Lead status starts 'nieuw'. ConvertLeadToProject (POST /leads/{id}/convert) is atomic: in one tx it sets lead.status='gewonnen' AND inserts a project (fase 'intake', status 'actief', lead_id back-link, company inherited) so you never get a won-lead-without-project or a duplicate project.

### Parallel workstream (opdracht) → project
Workstreams are a lighter engagement track. ConvertWorkstreamToProject (POST /workstreams/{id}/convert-project) has two paths: (a) target project already given → set workstream status='omgezet_project' + project_id link; (b) no project → tx creates a new project (naam/waarde/deadline inherited, samenvatting composed from doel+scope+bevindingen) and links it, atomically. Customer must match if a project is attached.

### Billing: quote → time entries → invoice → bunq payment → document
CreateQuote (POST /quotes) requires titel + ≥1 line; amounts server-computed; number LC-OFF-YYYY-NNNN allocated under advisory lock. UpdateQuoteStatus walks concept→verstuurd→geaccepteerd→afgewezen→verlopen; 'geaccepteerd' sets accepted_at and CANNOT be rolled back (409). CreateInvoiceFromQuote (POST /quotes/{id}/invoice) requires status='geaccepteerd' + ≥1 line; idempotent (returns existing non-cancelled invoice). CreateInvoice (POST /invoices) can also build from manual lines and/or selected billable, un-invoiced time_entry_ids (marks them 'gefactureerd'). UpdateInvoiceStatus walks concept→verstuurd→betaald→geannuleerd; 'verstuurd' sets sent_at, 'betaald' sets paid_at+paid_cents; a paid invoice cannot be reverted nor re-valued (legally final BTW record, 409). CreateInvoicePaymentRequestAction (POST /invoices/{id}/payment-request) queues a PENDING confirmation (toolName laventecareBetaalverzoekMaken) rather than calling bunq directly — approved via Settings/Telegram; RefreshInvoicePaymentStatus polls bunq RequestInquiry and marks 'betaald' on ACCEPTED/PAID/COMPLETED. Requires BUNQ_API_KEY/USER_ID/MONETARY_ACCOUNT_ID. GetInvoiceDocument (GET /invoices/{id}/document?format=html|ubl|json) renders branded HTML + UBL 2.1 (EN16931, InvoiceTypeCode 380, supplier CompanyID 88162710) + plain text and persists document_url/ubl_xml/generated_at.

### Document / dossier generation + AI advice
lc_documents is the seeded 26-item knowledge catalogue (visibility public/send_only/internal/contract, funnelStage, services). The frontend serves a 5-page react-pdf at api/laventecare/pdf/[key] (templateProfile proposal/discovery/legal/operations/service); PdfDossierContext passes CRM context via ?ctx querystring. CreateDossierDocument (POST /dossier-documents) records a generated PDF in lc_dossier_documents (company_id derived from lead/workstream/project). BuildDossierAdvice (GET /ai/dossier-advice, also AI tools laventecareKennisAdviesOpvragen + laventecareDossierCheckOpvragen — one store fn, two tool names) is deterministic and read-only: for exactly one target (company/lead/project/workstream) or a free query it builds requirements (customer_context, knowledge_catalog, commercial_scope, delivery_plan, governance, dossier_history) with status ok/attention/missing, scores knowledge docs by categorie/fase/markers, computes coverage %, and emits next_actions + evidence with overall status documentbasis_leeg/onvolledig/aandacht/klaar.

### The mail loop (outbound only)
GET /mailbox returns templates, outbox, inbox and conversation/thread data. SuggestMailContent (POST /mailbox/ai-suggest) builds a safe draft context (LaventeCare + agenda + rooster + notes + attachments + pilot-account block from account notes) and fills template variables via Grok with strict safety (never put passwords/secrets verbatim in customer mail); deterministic fallback if no API key. SendTemplatedMail (POST /mailbox/send-template) renders and sends via Microsoft Graph sendMail (max 6 PDF attachments, ≤3MB), driving outbox statuses sending/sent/failed/sent_unconfirmed, scrubbing pilot passwords at rest (MarkSent), and auto-logging an 'email' activity event. POST /mailbox/inbox-sync ingests Graph inbox messages idempotently; conversation_id groups inbound and outbound mail and the UI supports reply/read flows.

### Customer onboarding / portal access
After a project starts, AccessCredentials (POST /access-credentials, company_id required) store encrypted customer logins (AES-GCM) with environment 'pilot', sharing_policy 'veilig_kanaal'; the API exposes only secret_configured + secret_hint. company.portal_url/default_login_url/onboarding_status/data_processing_status track portal handover and GDPR processing state. Pilot login details flow into AI mail drafts via the server-side pilot-account block.

## Feature-oppervlak (per view)
- **Cockpit (GET /cockpit, AI tool laventecareCockpit)** — Aggregated dashboard: companies, contacts, accessCredentials, activeLeads/Workstreams/Projects (via isOpenStatus), actionItems, 5 each of open incidents/changes/decisions, documentCatalog, dossierDocuments, activity, mailbox summary, plus computed businessSignals and followUps. Counts via single round-trip. AI executor strips AccessCredentials before shipping to the external model.
- **Billing (GET /billing, AI tool laventecareBillingOpvragen)** — Commerce view: quotes, quote_lines, time_entries, invoices, invoice_lines + summary (quotes/open_quotes, billable_minutes, uninvoiced_minutes, open_invoices, outstanding_cents, paid_cents, default_provider 'bunq', bunq_ready). Optional companyId filter. Read tool instruction forbids creating invoices/payment requests here.
- **Mailbox (GET /mailbox + ai-suggest + send-template + template CRUD)** — Lists templates + outbox history; create/update templates (subject+body_html required); AI-suggest draft content (Grok); send templated mail via Microsoft Graph with PDF attachments.
- **Companies / Contacts / Access-credentials** — Full CRUD for the customer dossier, contacts (with primary-uniqueness), and encrypted portal logins. Company list returns derived counts; DeleteCompany is a GDPR cascade erasure.
- **Leads / Projects / Workstreams** — CRUD listers + create/update, plus conversions: /leads/{id}/convert (→project), /workstreams/{id}/convert-project, /signals/convert-lead. Workstream list excludes finished by default, sorts by deadline.
- **Quotes / Time-entries / Invoices** — Create quote, update quote status, quote→invoice, create time entry, create invoice (manual + time entries), invoice status, invoice document (HTML/UBL/JSON), payment-request (bunq, queued for confirmation), payment-refresh.
- **Governance: Decisions / Change-requests / SLA-incidents** — List + create + status-patch for each, all scoped to a project. Open-list filters differ per type (decisions show all; changes nieuw/beoordeeld/goedgekeurd; SLA open/in_behandeling/wacht_op_klant).
- **Documents / Dossier-documents / AI dossier-advice** — GET /documents (knowledge catalogue), GET /dossier-documents (generated PDF history), POST /dossier-documents (record a PDF), GET /ai/dossier-advice (deterministic coverage/recommendations). POST /documents/seed bulk-upserts the catalogue.
- **Activity timeline (GET/POST /activity)** — Customer-dossier timeline; creating contact/call/meeting/email events updates company.laatste_contact. All linked objects validated as same-company.
- **Action items (GET/POST/PATCH /actions)** — Follow-up tasks with polymorphic links; list shows only open/bezig/wacht_op_klant sorted by due_date; status freely patchable (e.g. 'done').
- **PDF viewer (frontend api/laventecare/pdf/[key] + documenten/[documentKey]/page.tsx)** — 5-page react-pdf rendering with templateProfile per document; alle PDF-bytes vereisen een eigen Clerk-ownercheck; een echte dossierlookup gebeurt alleen na ownercontrole en gebruikt de backendkey uitsluitend server-side.
- **Focus summary (GET /focus/summary, FocusBusinessStatus)** — Read-only tablet dashboard counting active leads/workstreams/projects, open+overdue actions, open quotes/invoices, outstanding_cents (SUM total-paid); raises attention tiles 'lc-overdue-actions' (high) and 'lc-open-invoices' (medium) linking to /laventecare.
- **Telegram /laventecare (/lc) + free-text + AI status** — Routes to the laventecare agent with lcPrompt (orchestrates cockpit/billing/klanten/opdrachten/kennisadvies/dossiercheck/dossierdocumenten + planning/notes); free-text routed via hasLaventeCareIntent/hasLaventeCareDossierIntent; AI cockpit/welcome surfaces live tool/agent counts and pending confirmations.

## Integraties
- AI agent 'laventecare' (🏢): 29 tools registered in ai/tools.go, gated by ai/agents.go Policies. 10 read tools (Cockpit, KennisZoeken, KennisAdviesOpvragen, DossierCheckOpvragen, KlantenOpvragen, ContactenOpvragen, LeadsOpvragen, ProjectenOpvragen, OpdrachtenOpvragen, ActiesOpvragen, DossierDocumentenOpvragen, BillingOpvragen — reads mapped to agents laventecare/dashboard/brain, Mutates:false) and 19 mutating tools (BetaalverzoekMaken, Klant Maken/Bijwerken, ContactMaken, Opdracht Maken/Bijwerken/NaarProject, Lead Maken/Bijwerken/NaarProject, Project Maken/Bijwerken, Actie Maken/Afronden, Besluit/ChangeRequest/SlaIncident Maken — agents laventecare/brain, Mutates:true, RequiresConfirmation:true → ai_pending_actions queue). KennisAdviesOpvragen and DossierCheckOpvragen share one executor case / BuildDossierAdvice (intentional alias).
- bunq payment integration: invoice payment requests created as queued confirmations (laventecareBetaalverzoekMaken) then executed on approval; RefreshInvoicePaymentStatus polls bunq RequestInquiry. Env: BUNQ_API_KEY, BUNQ_USER_ID, BUNQ_MONETARY_ACCOUNT_ID. payment_provider default 'bunq'. De huidige backend reserveert maximaal één createpoging per owner/factuur, gebruikt een stabiele bunq client-request-id en reconcileert op merchant reference; een ambigue write wordt niet automatisch herhaald.
- Microsoft Graph (branded LaventeCare client mailbox): draft/send with PDF attachments plus inbound Mail.Read sync, provider ids and conversation threading. Env: MICROSOFT_TENANT_ID/CLIENT_ID/CLIENT_SECRET, MICROSOFT_SENDER_EMAIL (default jeffrey@laventecare.nl), LAVENTECARE_MAIL_ENABLED. Configured-gate via laventeCareMailConfiguredFromEnv(). Distinct from the user's personal Gmail.
- Google (Gmail + Calendar): SyncGmail and SyncPersonalEventsDetailed (Telegram sync goroutines) feed emails and personal_events; buildBusinessSignals scans these for CRM-name matches, and briefings pull business-tagged calendar events. Gated on GoogleClientID/Secret/RefreshToken + GmailEnabled/GoogleCalendarEnabled.
- Telegram surfacing: /laventecare and /lc commands → laventecare agent + lcPrompt; routeFreeText intent detection; contextBriefingOpvragen scope 'laventecare' (embeds cockpit + BuildDossierAdvice); inline keyboard 🏢 button; AI status/welcome counts.
- Grok (xAI): drives mail-content suggestion (SuggestMailContent) with strict safety rules; deterministic fallback when no API key.
- Business-context wiring (Google round-trip): notes and personal_events carry business_context_type/_id/_title (laventecare | laventecare_company | _lead | _workstream | _project). Set by AI tools via engine/business_context.go inferLaventeCareBusinessContext (fuzzy-match score≥75) and frontend lib/laventecare/business-context.ts + lib/workspace-context.ts, which also encode [businessContextType/Id/Title] tokens into Google Calendar event descriptions so context round-trips through Google.
- Secret encryption: lc_access_credentials.secret_value_encrypted uses AES-GCM keyed only by the dedicated LAVENTECARE_SECRET_KEY; secrets are never returned by the API. Production requires a strong key that differs from app, bridge and intake credentials.
- Frontend (separate Next.js repo): consumes /api/v1/laventecare; serves react-pdf documents at api/laventecare/pdf/[key] and the dossier viewer; PDF bytes are owner-gated through the central HOMEAPP_OWNER_USER_ID check; dossier backend access uses X-API-Key only on the server.

## Historische gaten / opnieuw te verifiëren
- [OPGELOST 2026-07-17] De volledige owner-APIgroep, inclusief LaventeCare reads, staat achter apiKeyMiddleware. Alleen health is publiek; intake en bridge hebben ieder een eigen scoped credential. Frontend PDF-bytes hebben bovendien een Clerk-ownercheck.
- Status domains are mostly free-text with NO DB CHECK constraints. Only invoice (concept/verstuurd/betaald/geannuleerd) and quote (concept/verstuurd/geaccepteerd/afgewezen/verlopen) transitions are validated in Go (store/laventecare.go). company.status/relatie_type/contract_status/service_level/onboarding_status/data_processing_status, lead.status, project.fase/status, workstream type/status, action.status/action_type, change/decision/sla status are unvalidated — a typo (or an AI/UI persisting a value from the tool-description 'guidance' lists that doesn't exactly match) silently drops records from open-lists and the focus.go/cockpit active-status filters (NOT IN ('afgerond','gesloten',...)).
- Possible status name-drift: model/json elsewhere suggests 'verzonden'/'open' while the validated invoice enum uses 'verstuurd'. Verify handler/seed/frontend use exactly the store-layer canonical strings.
- Leads have no explicit 'verloren'/'gediskwalificeerd' close status or transition validation — only ConvertLeadToProject sets 'gewonnen'. There is no first-class way to mark a dead lead.
- laventecareKennisAdviesOpvragen vs laventecareDossierCheckOpvragen are framed differently ('advies' vs 'volledigheidscheck') but execute identical code — the distinction is cosmetic. Consider differentiating behavior or merging.
- [OPGELOST 2026-07-17] Microsoft Graph inbox-sync, lc_mail_inbox, conversation_id-threading, mark-as-read en replyflows zijn aanwezig. Mail.Read-consent/configuratie blijft een deploymentvereiste.
- GenerateInvoiceDocument writes document_url/ubl_xml even for concept (not-yet-sent) invoices with no status check — confirm generating legal UBL for drafts is intended.
- lc_decisions and lc_dossier_documents have no updated_at column — confirm no code writes updated_at to them.
- The deferred FK lc_time_entries.invoice_id (INITIALLY DEFERRED, added via DO-block) is absent on a fresh DB until ensureLaventeCareBillingSchema runs — confirm schema setup always precedes use.
- buildBusinessSignals matches CRM names against email/event/note text via normalized substring/term matching; short or generic company names could produce false-positive signals. Worth a minimum-length or word-boundary guard.
- Credential encryption uses the dedicated LAVENTECARE_SECRET_KEY. Production validates strength and separation from APP_SECRET_KEY, BRIDGE_API_KEY and LAVENTECARE_INTAKE_SECRET; backup and rotation procedures remain an operational owner action.
