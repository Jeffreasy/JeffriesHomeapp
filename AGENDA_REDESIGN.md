## Recommended direction
Direction A ("Dagrooster — agenda-first on mobile, grid on desktop") as the spine, hybridized with three grafts: (1) a thin month "heat strip" / dots row pinned above the agenda list on mobile, borrowed from the dots-only month concept of A's Maand tab but kept always-visible as a 1-row week/month navigator rather than a separate tab; (2) the desktop month-grid enrichment from A (taller rows, more chip room) but with the existing 300px GESELECTEERDE DAG side panel converted into the *same* AgendaListRow styling so mobile and desktop share one row component; (3) inline conflict + note rendering pulled up from SelectedDayEvent (lines 666-671) so conflict.message and note counts are visible in the row itself, never behind a title= tooltip. Net: phone defaults to a scrollable chronological agenda list with sticky day headers and full-width non-truncating rows; the 7-col grid is demoted to a navigator (dots) on mobile and stays the primary surface only at lg+.

## Why
This is a PWA where mobile is the primary surface, and the core defect is purely a mobile-layout problem: a 7-column grid physically cannot render timed event TEXT in ~50px cells, which is why titles truncate to "DKL: Af…" and chips wrap "07:00\nVroeg" with horizontal overflow. Direction A is the only one of the four that *removes* the constraint instead of negotiating with it — at full row width with text-sm the title gets ~28-30 chars on one line and line-clamp-2 catches the rest, so nothing truncates illegibly and the single-column list cannot overflow horizontally. It also matches the muscle memory of every shipping mobile calendar (Google/Apple/Fantastical agenda view), so there is zero re-learning. Critically for THIS codebase it is mostly additive and reuses primitives that already exist and are already exported: getDisplayEndDate, getTimeLabel, isMultiDay, formatDateRange (hooks/usePersonalEvents.ts), getEventTone/buildSingleDay/eventCoversDate/compareCalendarEvents/formatDaySummary (AgendaCalendar.tsx), conflictMap with ConflictInfo.message/level, and notesByEventId — so the new surface is a render change, not a data-model change. I grafted the dots strip from A's own Maand-tab idea (kept persistent as a heat row so users who think spatially never fully lose "how busy is this month" and get one-tap date jumping) and grafted the inline-conflict + shared-row-component ideas to kill the duplicated mobile-vs-desktop chip stacks in CalendarDayCell (lines 354-397), which is a net code reduction. The runner-up framings (forcing a legible grid, week-view-as-default, bottom-sheet day drill-down) were rejected because: a legible 7-col grid is geometrically impossible at 380px; week view still needs 7 columns and only shows 7 days so paging is heavy; and a pure bottom-sheet drill-down hides the day content behind an extra tap, which is worse than a list for the common "what's happening next" question. The amber primary (#f59e0b) is reserved for notes/accent so the indigo(dienst)/emerald(afspraak)/sky(wachtrij)/rose(conflict) semantic palette stays unambiguous.

## Risks
- Sticky stacking math: the page already has a sticky header (page.tsx line 391, top-0) AND a fixed mobile bottom tab bar — the new sticky day headers need a correct top offset (--agenda-sticky) and the list needs pb-safe, or headers will hide behind the page header / the last row behind the tab bar.
- Two mobile render paths (list + dots grid) to keep in sync; buildAgendaList + AgendaListRow is ~120 LOC net-new even though it reuses the data layer.
- Multi-day events appear once on their start date in a list (vs spanning grid cells), so they can be missed — mitigated by the isMultiDay + formatDateRange 'loopt door t/m …' affordance in the row meta; verify against real overnight diensten (the conflict layer already rolls night shifts past midnight).
- The user edits these files live and concurrently (per the agenda-review memory note) — re-read AgendaCalendar.tsx and page.tsx immediately before editing and use targeted git add, not git add -A, to avoid clobbering in-flight changes.
- Behavioural default ('agenda' under sm) must be CSS-gated (sm:hidden / hidden sm:block), not JS width detection, or SSR/hydration will mismatch — extending CalendarMode to include 'agenda' is fine but the mobile list should render regardless of mode via CSS.
- line-clamp-2 still has a ceiling: a pathologically long title wraps to 2 lines then ellipsizes — acceptable (full text is one tap away in the edit modal) but worth a row min-h so 1-line and 2-line rows don't jitter.
- Reserving amber for notes means conflict text uses amber-300 too (matching the existing SelectedDayEvent treatment) — keep the conflict's PRIMARY signal the rose border/dot/AlertTriangle so amber-as-note vs amber-as-conflict-text doesn't read ambiguously.


## The four directions explored

### Dagrooster — agenda-eerst op mobiel, raster op desktop (effort: medium)
A 7-column month grid is a desktop information structure; forcing it into 380px is the entire problem — text must shrink to 9px, titles truncate to "DKL: Af…", chips wrap "07:00\nVroeg", and the grid still overflows horizontally. Direction A stops fighting that. On a phone the calendar's job is "what's happening, in order, legibly" — so the DEFAULT mobile surface becomes a vertically scrolling chronological AGENDA LIST: sticky day headers + full-width event rows where NOTHING truncates (full title, full time range, dienst/afspraak distinction, conflict + note badges all visible). The month grid is demoted to a compact dots-only overview that you can pop open as a bottom sheet to jump dates — it answers "how busy is this month" and "take me to that day", not "read event text". Desktop is untouched in spirit: it keeps (and gets a richer) month grid + the 300px GESELECTEERDE DAG side panel, because at lg/xl there is room to render chips legibly. One component, one data model, two presentations gated purely on the existing sm/lg breakpoints.

### Compact Month, Day Drill-Down (effort: medium)
A month grid is the right mental model for "where am I this month?" but text chips are the wrong payload for a 50px phone cell. So we keep ONE month grid at every breakpoint and change only what lives inside the cell: on mobile each day becomes a date number plus a row of up-to-4 colored dots (dienst/afspraak/wachtrij/conflict/notitie) with an optional overflow count — never text. The grid stays glanceable and never overflows horizontally. All actual reading happens in a drill-down: tapping a day on mobile opens a bottom sheet with the full event list; on desktop the day's text chips stay in the cell AND the existing 300px GESELECTEERDE DAG sidebar remains the detail surface. One grid, two detail strategies keyed off the breakpoint. This directly kills the three current mobile failures (wrapping "07:00\nVroeg", truncating "DKL: Af…", and the horizontal scrollbar) because nothing inside a mobile cell is text anymore.

### Week-first responsive (vertical stacked-day week list on mobile) (effort: high)
A 7-column grid cannot legibly render timed text on a ~380px phone — that is the root cause of every truncation in the current mobile month view. Direction C inverts the default: on a phone the primary surface is a VERTICAL stacked-day WEEK list (7 day-rows stacked top-to-bottom, each row full-width ~340px), where every event chip gets a full line of horizontal room and NOTHING truncates. The horizontal 7-col grid is demoted to a secondary, dots-only "month overview" used purely for navigation/drill-down. Desktop keeps both month and week as a real grid (it has the width), but week becomes a proper time-rail. The core bet: people on a phone overwhelmingly ask "what's my week / what's today" — a scrollable week of legible rows answers that better than a cramped month matrix. This reuses the existing CalendarMode union, buildCalendarDays (already builds a 7-day week array), getEventTone, and the SelectedDayPanel logic, so it is an additive restructure rather than a rewrite.

### Refined Grid (effort: medium)
Don't change the paradigm — fix the craft. The 7-column month grid stays on both desktop and mobile, but every legibility failure (wrapping chips, illegible truncation, horizontal overflow, cramped rhythm, weak dienst-vs-afspraak contrast) gets engineered out. The discipline is: a month cell never tries to be a schedule. It shows at most a few single-line chips plus a clean "+N", and the SelectedDayPanel is the full-detail surface. This is the Fantastical/Linear move: the grid is a dense overview that you read by color and density, and you drill into a day for the words. Mobile keeps the grid but trades chips-with-text for an event-density dot rail under each day number, because timed text cannot be legible in a 50px cell — the existing mobile SelectedDayPanel summary block (already in the header, lines 216-240) becomes the primary read surface and gets promoted.
