## Verdict
Today the Mail feature looks polished on the surface (branded HTML shell, sandboxed preview, send-readiness checks, password redaction at-rest) but is NOT yet trustworthy or professional where it counts: the actual client-facing artifact and the irreversible action are both unguarded. Two structural gaps undermine the whole thing. (1) Correctness/trust: every merge value — CRM fields, the free-text key=value editor, AI/Grok drafts, PDF-extracted text — is substituted into the SENT email HTML with zero escaping (renderTemplate at store/laventecare_mailbox.go:2465 does raw strings.ReplaceAll; the preview mirror at view:1033 does the same). A company name like 'Jansen & Zn' or a workstream 'Scope <fase 2>' silently corrupts the mail a real client receives, and an AI/PDF-injected '<img onerror=...>' becomes live markup in a client inbox AND is persisted at-rest. The sandboxed preview iframe (view:536) hides this from the owner — the preview looks fine, the sent mail does not. (2) Friction inversion: emailing a real client — the highest-consequence, irreversible action in the app — fires on a single click with no confirmation and no recipient echo (view:577), while resolvedEmail can silently fall back to selectedContact.email (view:157). It has less friction than deleting a note. Layered on top, the 'mailbox' is really two disconnected logs: inbox and outbox never join on conversation_id (store:837 / mailOutboxSelectSQL never selects it / model:556-557 falsely claims they're threaded), and both row types are read-only dead ends — no open, no resend of a failed send, no reply, no mark-as-read (view:713, view:732). SINGLE BIGGEST WIN: build the read-only HTML preview MODAL the owner asked for — a full-fidelity, escaped rendering of exactly-what-gets-sent — and make it the shared surface for (a) the pre-send confirmation, (b) opening any outbox row, and (c) opening any inbox message. One modal component fixes the owner's stated pain, adds the missing send-guard, and turns the dead ledgers into a real mailbox. Pair it with the one non-negotiable backend fix (escape-on-render) and Mail jumps from 'looks nice, scary to use' to genuinely professional.

**Biggest win:** Build the read-only HTML preview modal the owner flagged, as one reusable component, and wire it four ways: (a) full-fidelity escaped preview of exactly-what-gets-sent, (b) the pre-send confirmation with recipient echo, (c) opening/resending any outbox row, (d) opening/replying to any inbox message — paired with the one non-negotiable backend fix of escaping every substituted value on the HTML render path (store/laventecare_mailbox.go:2465 + the view:1033 mirror) so the modal shows the true artifact.

## Themes
- Trust the artifact, not the preview: the sandboxed iframe (view:536) lulls the owner into trusting output that the unescaped sent-HTML path (store:2465, view:1033) does not match. Every fix must make preview == exactly-what-is-sent, on one shared escaped render path.
- Friction must scale with consequence: the irreversible client-send is one click (view:577) while reversible drafts are also one click — invert it so the high-stakes action gets a confirmation gate and the safe one stays frictionless.
- All external/AI/PDF input is untrusted by default: CRM fields, the free-text editor, Grok drafts (handler:2613) and PDF extracted_text converge on the same render sink, so escaping at the sink (item 1) is the single chokepoint that fixes XSS, broken-render, and prompt-injection at once — don't scatter input-side sanitizers.
- The conversation is the product: a client mailbox that shows two disconnected logs (store:837, model:556-557) and dead read-only rows (view:713/732) isn't a mailbox. Threading by conversation_id and making rows actionable is what separates 'send/receive log' from 'professional mailbox'.
- Reuse one surface: a single read-only escaped preview-modal component serves the owner's ask, the send-confirmation, outbox-open/resend, and inbox-open/reply — build it once, wire it four ways.

## Solid
- The branded HTML shell is built correctly from structured fields using escapeMailText/escapeMailAttr (store:2442-2453, 2457-2463) — the escaping discipline exists, it's just not applied to the dynamic substitution path. The fix extends an existing pattern rather than inventing one.
- pilot.access_block_html, the one legitimately-raw HTML slot, is reset to '' then rebuilt entirely from per-field-escaped values via mailAccessFieldHTML (store:2201-2227) — a correctly-scoped trusted-HTML boundary that the escape-on-render fix can simply allow-list.

**9. Template CRUD accepts arbitrary body_html with no shell/structure validation**  
- _Backend correctness & security (send, AI, render, inbound)_ * `consistency` * `backend/internal/handler/laventecare.go:193`
- **Impact:** CreateMailTemplate/UpdateMailTemplate (handler 193,211) store body_html as-is (store CreateMailTemplate:607-624 only TrimSpace + non-empty check).
